# 项目面试讲解指南

本文档用于准备职途 Agent 的项目介绍、技术追问和产品取舍问题。讲解时应明确项目是 MVP / prototype，不虚构用户量、转化率或商业结果。

## 1. 项目背景

求职者通常会使用不同工具完成 JD 分析、能力评估、学习规划、项目准备、简历修改和模拟面试。这些步骤各自独立，用户需要重复输入背景，也难以知道当前进度和下一步。

职途 Agent 尝试将这些步骤组织成一条连续的 AI Agent Workflow，让每个阶段的结构化结果成为后续阶段的输入，并最终转化为可以执行和跟踪的任务。

## 2. 为什么做这个项目

- 求职准备是一个天然的多阶段决策与执行场景
- 单次聊天很难持续保存岗位、能力、项目和面试上下文
- 适合验证“AI 生成 + 确定性规则 + 状态管理”的组合方式
- 可以同时展示产品设计、Agent Workflow、前端工程和 AI API 集成能力

项目目标不是替代招聘顾问，而是验证如何降低求职者从理解岗位到开始行动的组织成本。

## 3. 用户痛点

1. JD 信息密集，不容易识别真正重要的能力。
2. 用户知道自己“有差距”，但不知道差在哪里、先补什么。
3. 学习、项目、简历和面试之间缺少连接。
4. AI 可以生成大量建议，但建议通常不能直接执行。
5. 用户缺少统一进度和下一步反馈。

## 4. 产品核心链路

```text
目标岗位 / JD
→ JobProfile
→ AbilityProfile
→ GapReport
→ Roadmap
→ ProjectRecommendation
→ ResumeReport
→ InterviewReport
→ Tasks
→ Dashboard / Profile
```

核心设计原则是：

- 每个阶段有明确输入和结构化输出
- 后续节点复用前序结果，减少重复输入
- AI 负责非结构化理解和个性化生成
- 规则负责可预测、可测试的计算
- 最终输出不是报告集合，而是可跟踪任务与进度反馈

## 5. 技术架构

前端使用 React 19、TypeScript、TanStack Router / Start 和 Tailwind CSS。

状态集中在 `AgentState`：

- 页面使用 `loadState` / `saveState`
- 数据持久化到浏览器 `localStorage`
- Dashboard、Profile 和 Tasks 都读取同一份状态

AI 和搜索采用服务端代理：

- 浏览器调用 TanStack Start server function
- server function 构建 Prompt 并校验输入
- `.server.ts` 模块读取服务端环境变量
- SiliconFlow 提供 LLM，Tavily 提供搜索

当前没有登录、真实数据库或多设备同步。

## 6. AI 能力

LLM 用于：

- 从岗位名或 JD 生成结构化岗位画像
- 根据用户经历和岗位画像生成能力画像
- 生成三档项目推荐
- 生成简历优化报告
- 生成五类模拟面试题
- 评分用户回答并生成优化答案
- 对 PDF 全文进行知识问答

Prompt 要求模型返回约定 JSON，客户端业务层负责解析、标准化和保存结果。

需要准确说明：

- 当前没有复杂的多 Agent 编排框架
- Workflow 主要由页面流程、统一状态、结构化数据和服务端 AI 操作组成
- AI 输出可能不稳定，当前尚未建立生产级评测和监控

## 7. 安全整改

### 原问题

如果私密 API Key 使用 `VITE_` 环境变量或在浏览器中直接调用第三方 API，Key 可能进入客户端构建产物或被用户查看。

### 当前方案

- SiliconFlow 和 Tavily 调用迁移到 server function
- 服务端模块通过 `process.env` 读取 Key
- 客户端只传业务参数并接收业务结果
- `.env` 被 Git 忽略
- Tavily 数据在服务端转换为有限的安全字段

### 仍需补充

- 认证和授权
- 调用限流
- 审计与监控
- 密钥轮换
- 输入长度、文件大小和请求频率限制
- 用户隐私与数据删除机制

因此当前完成的是 API Key 暴露风险整改，不是完整生产安全体系。

## 8. fallback 设计

外部模型可能因为网络、余额、格式异常或服务不可用而失败。为了避免整个主链路完全中断，以下模块具备本地 fallback：

- 项目推荐
- 简历报告
- 模拟面试题
- 面试回答评分

fallback 使用前序结构化数据生成基础结果，保证结果仍与目标岗位和差距相关。

取舍：

- 优点：演示稳定、可测试、降低单点依赖
- 缺点：规则内容的个性化和表达丰富度低于正常模型结果
- 边界：岗位画像与能力画像目前仍依赖 AI，不是完整离线模式

## 9. 项目难点

### 难点一：让多个 AI 功能共享上下文

解决方式：使用统一 `AgentState` 保存结构化结果，每个节点只读取需要的数据，避免依赖长对话历史。

### 难点二：控制 AI 输出结构

解决方式：在 Prompt 中定义严格 JSON 字段，对返回结果进行解析、标准化，并为部分模块提供 fallback。

### 难点三：把建议转化为执行

解决方式：从 Roadmap 的每日任务和周目标、项目交付物、面试练习建议生成稳定 ID 的 `AgentTask`，并保留用户已修改的任务状态。

### 难点四：Dashboard 展示真实数据

解决方式：指标全部从 `AgentState` 和 Tasks 派生，不使用写死的进度或虚构数据，并根据当前缺失步骤推导下一步动作。

### 难点五：保护第三方 API Key

解决方式：将外部调用迁移至 server function 和服务端模块，避免私密 Key 进入浏览器。

### 难点六：控制 MVP 范围

项目没有在缺少真实用户验证时提前加入登录、数据库、多租户和复杂 Agent 框架，而是优先完成并验证核心 Workflow。

## 10. 如果继续迭代

### 第一阶段：验证产品价值

- 进行目标用户访谈和可用性测试
- 验证 Workflow 是否减少重复准备和决策成本
- 记录各节点的放弃率、重复生成率和任务完成情况
- 调整输入字段、报告长度和节点顺序

### 第二阶段：改善数据能力

- 增加数据导入、导出和版本迁移
- 在确认多用户需求后加入认证、数据库和云端同步
- 支持多个目标岗位和不同版本报告

### 第三阶段：升级 AI 与 RAG

- 建立 Prompt 版本和模型输出评测
- 增加超时、重试、限流和可观测性
- 将 PDF 全文问答升级为文档分块、Embedding、检索、重排序和引用溯源
- 根据成本和质量需求选择模型路由

### 第四阶段：生产化

- 权限和隐私治理
- 用户数据删除和导出
- 内容安全和敏感信息处理
- 服务端审计、告警和密钥轮换

## 常见追问简答

### 为什么不用数据库？

当前目标是验证单用户本地 Workflow。使用 `localStorage` 可以减少非核心基础设施，但代价是无法登录、同步和可靠备份。确认真实多用户需求后再引入数据库更符合 MVP 范围控制。

### 为什么 Gap 和 Roadmap 不全部交给模型？

这些节点依赖明确的输入和可解释规则。使用确定性逻辑更稳定、便于测试，也减少调用成本。模型更适合处理岗位文本理解和表达生成。

### 这是多 Agent 系统吗？

当前不是复杂的自治多 Agent 系统。它是一个由多个 AI 操作、规则节点和共享状态组成的 Agent Workflow 产品原型。这样描述比宣称多 Agent 更准确。

### RAG 是怎么做的？

PDF.js 提取全文并保存在 `localStorage`，提问时将全文和问题一起发送给模型。当前没有向量数据库、Embedding 或检索，因此是全文知识库问答原型，不是完整 RAG。

### 如何证明不是写死数据？

Dashboard、Profile 和 Tasks 都从 `AgentState` 派生；任务状态变化会影响统计和下一步动作。核心规则还覆盖了单元测试和构建验证。
